# Ask는 근거가 확실할 때만 보여준다 — 불확실하면 추측 대신 비운다

Ask 답변의 출처(sources)와 "못 찾음" 응답을 두 곳에서 결정해야 했다: (1) 모델이 실제로 어떤 청크를 근거로 썼는지 확신할 수 없을 때 출처란을 무엇으로 채울지, (2) 검색 점수가 낮아 근거가 약해 보일 때 그래도 답을 만들지. 두 경우 모두 **"확실하지 않으면 그럴듯한 값으로 채우지 않고 비워둔다"** 는 같은 원칙으로 결정했다. 근거로 삼은 청크는 모델이 답변 끝에 붙이는 숨김 인용 태그(발췌 순번)로만 판정하며, 태그가 없거나 파싱에 실패하면 출처는 상위 청크로 대체하지 않고 빈 배열로 둔다. 검색 결과 1등 점수가 문턱 아래면 LLM을 아예 호출하지 않고 "저장된 자료에서 못 찾았다"로 조기 반환한다.

## Considered Options

- 출처: 태그가 없으면 상위 N개 청크로 대체(기존 동작) — 화면은 항상 출처를 보여주지만, 진짜 근거인지 알 수 없는 값을 근거인 것처럼 보여줘 `CONTEXT.md`의 "Ask to Recall"(근거가 된 Chunk가 출처로 따라온다) 정의를 위반한다.
- 게이트: 항상 LLM을 호출하고 낮은 점수의 청크도 그대로 근거로 씀 — 관련 없는 청크로 그럴듯한 답을 지어낼(환각) 위험을 그대로 둔다.

## Consequences

- `citedChunkIds`가 비면 `sources: []`이고, 사용자는 "이번 답엔 출처가 안 붙었다"만 본다 — 틀린 출처가 붙는 것보다 안전하지만, 태그 파싱이 자주 실패하면 출처가 자주 비게 된다. 태그 파싱 성공률은 이번 스펙의 평가 범위 밖(온디바이스 LLM 생성은 노드에서 자동 채점 불가)이라 수치로 추적하지 못한다.
- 게이트 문턱(0.3, 실측 후 조정)보다 낮은 질문은 LLM 호출 없이 즉시 "못 찾음"으로 끝난다 — 확장 쿼리를 포함해 병합·정렬된 결과의 1등 점수 기준이라, 확장 쿼리가 찾아낸 근거는 그대로 살아남는다.
