# 빈 방 해결: 안전장치 붙인 히스토리 prefill

신규 사용자의 빈 메모리(cold-start) 문제를 풀기 위해 설치 시 브라우저 히스토리에서 URL을 골라 다시 받아 캡처하는 Prefill을 v1에 넣는다. 단독 제품이라 라이너 하이라이트 같은 외부 씨드가 없고, "오래될수록 가치"인 제품은 첫날 메모리가 비어 활성화 지표("1주 내 첫 회수")를 달성하기 어렵기 때문이다. 히스토리 prefill은 프라이버시 위험이 크지만, 아래 안전장치로 "공개적으로 읽을 수 있던 것만" 담도록 한정한다.

## Considered Options

전체 히스토리 자동 prefill은 원래 반대했다(ADR 검토 과정). 히스토리는 URL만 있고 내용이 없어 re-fetch가 필요하며, 은행·웹메일·로그인 페이지 등 denylist가 막으려는 민감 페이지가 대부분이기 때문이다. 대안은 "열린 탭만 가져오기 + going-forward 캡처/하이라이트"였다. 그러나 안전장치를 붙이면 히스토리 prefill이 cold-start를 더 강하게 해결한다고 판단해 채택했다.

## Consequences

- **denylist 적용:** prefill 대상에서 알려진 민감 도메인을 거른다.
- **쿠키 없이 re-fetch(credentials: omit):** denylist가 모르는 로그인 페이지는 공개(로그아웃) 버전만 돌아오고, 그 로그인 벽은 thin-page 게이트가 버린다. 사적 내용은 들어오지 않는다.
- **기간 설정:** 최근 7/30/90일/직접 지정, 기본 30일.
- **예상 시간:** 시작 전 URL 수로 대략 추정, 진행 중 실제 처리 속도로 라이브 ETA 갱신.
- **큐 관리:** prefill 백필은 라이브 캡처보다 후순위(라이브가 없을 때 빔). re-fetch 동시성 3~5로 제한, 재개·취소 가능, 정규화 URL로 중복 방지(idempotent).
- **권한 비용:** `history` 권한("브라우징 기록 읽기" 경고)과 `<all_urls>` host 권한("모든 사이트 데이터" 경고)이 필요해진다. 온보딩에서 이유를 정직히 설명한다.
- 이 기능은 원래 1-pager 대비 v1에 가장 크게 추가된 것으로, 스코프가 그만큼 늘어난다.
